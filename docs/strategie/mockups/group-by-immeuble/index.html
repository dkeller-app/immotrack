🏛 Tri visuel par immeuble — onglet IRL

Mockups de 4 variantes UX · sujet UX-GROUP-BY-IMMEUBLE · 2026-05-17
Demande user : « pour l'onglet IRL, tous les biens sont mélangés. On peut faire un tri visuel par immeuble avec des séparateurs ? règles pour tous les onglets »
Décisions captées : groupement par immeuble seul · headers statiques (pas de sticky) · compteur = nb lots · groupe « (Sans immeuble) » en dernier.
Choix à faire : variante A / B / C / D (ou hybride). La variante validée sera codée en v15.76 sur l'onglet IRL réel, puis propagée onglet par onglet au fil de la roadmap V3.

Variante A — Lignes sobres

Séparateur fin avec nom immeuble + compteur. Minimaliste, KISS, zéro distraction visuelle. Le bien reste la star.
+ Pour : densité élevée, ultra rapide à lire, dark mode élégant, responsive trivial, peu de CSS. Mobile OK.
− Contre : pas de KPI immeuble (loyer total, occupation), pas d'action sur le groupe entier, peut se perdre si beaucoup de lots par immeuble.

Variante B — Cards riches (KPI immeuble)

Header en bloc card avec icône, adresse, compteur lots + KPI immédiats : loyer total, alertes (DPE F/G, IRL manquants).
+ Pour : contexte immeuble immédiat (loyer total, alertes), différenciant vs Rentila/BailFacile, donne envie de cliquer sur l'immeuble.
− Contre : densité réduite, KPI à calculer (perf si N gros), responsive plus délicat, écrase l'œil sur petits écrans.

Variante C — Bandeau sticky (suit le scroll)

Header qui colle en haut quand on scroll dans son groupe : on sait toujours dans quel immeuble on est. Compteur affiché en permanence.
+ Pour : repérage permanent en scroll long (utile gros patrimoine), pattern moderne (Stripe, Linear), élégant sur desktop.
− Contre : pénible en responsive mobile (header bouffe l'écran), z-index conflicts possibles avec topbar, backdrop-filter coût perf, complexifie le CSS de l'onglet.

Variante D — Sections collapsibles (accordéon)

Chaque immeuble est un bloc qu'on peut replier/déplier. Compteur visible, badge d'alerte si pending. Idéal pour focaliser sur un immeuble à la fois.
+ Pour : très lisible avec 5-10+ immeubles, l'utilisateur replie ce qu'il a déjà traité, état persistable (localStorage), bonne UX mobile.
− Contre : 1 clic supplémentaire pour voir le contenu, pas adapté si l'utilisateur veut tout voir d'un coup (impression), animation collapse à gérer.
